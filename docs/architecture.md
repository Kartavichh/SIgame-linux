# Архитектура проекта

Документ накапливает принятые архитектурные решения. Обновляется по мере

## B3. Графический интерфейс — РЕШЕНО

**Выбран Tauri.**

- Ядро и вся игровая логика — на Rust.
- Внешний вид (окна, табло, кнопки, редактор) — на веб-технологиях (HTML/CSS/JS),
  которые Tauri отображает.

**Почему:**
- Видео и изображения в вопросах работают «из коробки» (теги `<video>`/`<img>`),
  без ручного декодирования кадров.
- Редактор паков (формы, перетаскивание, превью медиа) делать значительно проще.
- Требование «на Rust» сохраняется: логика игры в Rust, веб только отображает.

**Принятый компромисс:** придётся освоить минимум веб-технологий во фронтенде.
Это сочли приемлемым.

## Отдача медиа в клиенте — РЕШЕНО (Этап 2)

**Проблема:** медиаплеер WebKitGTK (на Linux это GStreamer) не умеет
воспроизводить `<video>`/`<audio>` по кастомной схеме `asset://`, которую
предлагает Tauri (`convertFileSrc`). Картинки грузятся (их декодирует сам
WebKit), а видео/аудио падают с `MediaError code 4 (SRC_NOT_SUPPORTED)`.

**Решение:** клиент поднимает встроенный HTTP-сервер на `127.0.0.1:<случайный порт>`
(крейт `tiny_http`), который отдаёт медиа из временной папки с правильным
`Content-Type` и поддержкой заголовка `Range` (перемотка). Фронтенд формирует
URL вида `http://127.0.0.1:<порт>/<файл>`. GStreamer такой источник понимает.

Замечание по окружению: у пользователя `proxychains` настроен с
`localnet 127.0.0.0/...`, поэтому обращения к localhost не уходят в прокси.

## B1. Тип ПО — РЕШЕНО (уточнено: headless-сервер)

**Клиент-сервер с раздельными бинарниками.** Требование друга: игру нужно уметь
запускать и держать на **headless-сервере** (без GUI, например на домашнем
сервере по SSH). Поэтому сервер и клиент — это два независимых бинарника, а не
один в двух режимах:

- **`sigame-server`** — headless-бинарь, **без GUI**. Запускается в терминале
  (флаги: порт, путь к паку). Слушает сеть, хранит **авторитетное** состояние
  игры (`core::game::Game`), следит за соблюдением правил. Управление —
  аргументы запуска + простые CLI-команды (показать игроков/состояние, выключить).
  Может работать на машине без монитора.
- **`sigame-client`** (Tauri GUI) — подключается к серверу по сети. У клиента есть
  **роль**:
  - **игрок** — видит табло, жмёт кнопку ответа, видит свои очки;
  - **ведущий** — видит правильные ответы, открывает вопросы, судит
    «верно/неверно», управляет ходом партии.

**Роль ведущего на headless-сервере — РЕШЕНО:** ведущий судит из **GUI-клиента**
(режим «ведущий»), подключаясь к серверу с любого устройства. Сам сервер решений
по игре не принимает — он лишь принимает команды от привилегированного клиента
(ведущего) и рассылает состояние остальным. Так удобно судить устные и
медиа-ответы, а сервер при этом остаётся без экрана.

Почему так, а не «CLI добавить в общее приложение»: разделение на отдельные
бинарники у нас уже заложено структурой воркспейса (крейты `server` и `client`),
поэтому это самый естественный путь — просто «оживляем» крейт `server`.

P2P (опционально по требованиям) — откладываем на будущее. Ядро отделено от сети,
поэтому переход к P2P позже не потребует переписывания логики.

## B2. Структура проекта — РЕШЕНО

Workspace из нескольких крейтов; UI-часть оформлена как Tauri-приложение:

```
sigame-rs/
├── core/        # модель паков + правила игры (чистый Rust, без UI и сети)
├── server/      # headless игровой сервер: бинарь sigame-server (использует core)
└── client/      # Tauri-приложение (игрок / ведущий)
    ├── src-tauri/   # Rust-часть клиента (использует core)
    └── frontend/    # веб-интерфейс: табло, игра, редактор паков
```

Редактор паков на старте — часть клиентского приложения (отдельная вкладка/режим),
а не отдельная программа.

`server` — самостоятельный headless-бинарь (см. B1), а не библиотека, встроенная
в клиент. Клиент к нему подключается по сети так же, как и любой другой игрок.
