# Сетевой слой

Решения раздела E из `TODO.md`. Реализовано на Этапе 5.

## E1. Транспорт и протокол — РЕШЕНО

**TCP + строки JSON.** Одно сообщение = одна строка JSON, оканчивающаяся `\n`.
Тип сообщения — в поле `"type"` (тегированные перечисления serde).

**Почему не WebSocket** (хотя сначала планировался):

- **Ноль внешних зависимостей** — TCP и потоки есть в стандартной библиотеке.
  WebSocket потребовал бы внешний крейт. Так проект собирается без интернета.
- **Легко отлаживать** — сообщения человекочитаемы, можно подключиться вручную.
- **Сервер остаётся «худым»** — headless-серверу не нужен веб-стек.

Фронтенд (JS) не умеет открывать «сырой» TCP, поэтому **Rust-часть клиента
работает «трубой»**: читает строки из сокета и пересылает их фронтенду событием
Tauri `net:message`, а команды от фронтенда (`net_send`) пишет в сокет. Вся
игровая логика интерфейса — в JS.

### Сообщения клиент → сервер (`ClientMsg`)

| type        | поля                     | кто шлёт           |
|-------------|--------------------------|--------------------|
| `hello`     | `name`, `host` (bool)    | все, первым делом  |
| `start`     | —                        | ведущий            |
| `pick`      | `theme`, `question`      | текущий выбирающий |
| `buzz`      | —                        | игрок              |
| `judge`     | `correct` (bool)         | ведущий            |
| `reveal`    | —                        | ведущий            |
| `next_round`| —                        | ведущий            |

### Сообщения сервер → клиент (`ServerMsg`)

- `welcome` — `{ id: Option<PlayerId>, host: bool }` (id есть у игрока, у ведущего `null`).
- `state` — полный снимок состояния (`Snapshot`): фаза, игроки со счётом и флагом
  онлайн, табло (темы × клетки с признаком «сыграно»), текущий вопрос.
  **Правильный ответ (`current.answer`) есть только в снимке для ведущего.**
- `error` — `{ message }`.

Сервер шлёт **снимок целиком** при каждом изменении — это проще и надёжнее
инкрементальной синхронизации (нет рассинхрона), данных немного.

## E2. Подключение игроков — РЕШЕНО

Клиент соединяется по `адрес:порт`. Сервер слушает `0.0.0.0:<порт>` (по умолчанию
`7777`), поэтому доступен и из локальной сети. Первое сообщение — `hello` с именем
и ролью. Авто-поиск в сети и коды комнат — на потом (v2).

## E3. Никнеймы и переподключение — РЕШЕНО

- **Имя = личность.** Сервер хранит соответствие имя → `PlayerId`.
- **Обрыв связи** не удаляет игрока из партии — его счёт сохраняется, в снимке он
  помечается как офлайн.
- **Повторный вход под тем же именем** возвращает того же игрока (его `PlayerId`).
  Это работает в любой фазе игры (а вот *новый* игрок может зайти только в лобби —
  движок не добавляет игроков после старта).
- **Дубликаты отклоняются:** второй ведущий — «ведущий уже подключён»; второе
  подключение под именем, которое сейчас онлайн — «игрок с таким именем уже в игре».

## Архитектура исполнения на сервере

- Один бинарь `sigame-server`, **поток на подключение**.
- Всё состояние — за одним мьютексом `Arc<Mutex<Hub>>`. Команды применяются строго
  по очереди, поэтому «кто первый нажал» определяется порядком получения сервером.
- Консоль сервера: `status`, `players`, `start`, `help`, `quit`.

## Медиа по сети — РЕШЕНО (начало Этапа 6)

Сервер раздаёт медиа пака по HTTP на отдельном порту (`--media-port`, по умолчанию
`порт_игры + 1`), с поддержкой `Range`. Порт сообщается клиенту в `welcome`
(`media_port`). Клиент строит URL `http://<сервер>:<media_port>/<файл>` и
подставляет его в `<img>/<video>/<audio>`. Реализация: `server/src/media.rs`
(та же схема, что в локальном просмотре на Этапе 2 — WebKitGTK не играет медиа по
`asset://`, поэтому только HTTP).
